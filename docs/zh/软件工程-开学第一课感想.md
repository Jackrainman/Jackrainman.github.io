## 软件工程开学第一课感想

说实话，上软件工程课前，我心里想的就是一门理论课，也不一定会有很多收获。但听完第一节课，发现跟我平时在实验室干活碰到的那些事儿对上了，感觉就不太一样了。

我在机器人实验室做电控，平时写代码经常踩坑。比如版本控制这事，之前有过四个人搞出六个版本，最后只有一个能用，剩下全废，精力时间全浪费。后来又有个同学不小心把战队正常代码覆盖了，还好用git回滚救回来了。从那以后，git 和 GitHub 就成了每天必用的工具。还有编码风格，以前觉得能跑就行，后来发现团队协作的时候，别人看不懂你的代码，跟没有一样，甚至自己都不记得自己的代码是什么意义，在优化的时候也一头雾水。所以课上讲这些，我听着特别有感触。

课上讲到结对编程，说现在AI能帮上忙。我也在用，比如在VSCode里敲代码，AI能帮我回忆旧代码是干啥的，提醒我哪些地方容易踩坑，还能把那些看不懂的底层代码讲明白，讲得挺细，像老师一样。但我发现一个事：不能让它自己乱发挥。有次让它自由写，后来代码我开着 git 对比都认不出来了，又得回滚。AI毕竟有自作聪明的时候，出了错也解释不清。所以工具是工具，怎么用它还得自己拿得准。

课下我也经常会看一些开源的项目，star 数量多的都挺有意思，即使很少也能说明有更多的人为了软件平权在做努力。像Linux内核那种项目，你提个issue，真有懂的人愿意帮你弄，纯粹是出于兴趣。新人看大佬的代码能学到东西，大佬也能通过这个积累点影响力，最后大家都能用上更好的软件。这种模式确实把很多人的力量凑到一块了。最近还看到OpenClaw、OpenCode这种重量级工具，也都是社区里冒出来的，实用又强大。

第一节课听完，我原来的想法就变了。软件工程不是什么空的理论，是人家把踩过的坑总结成了一套方法。不管是自己跟自己协作（比如版本控制、代码风格），还是跟AI配合，或者参与开源社区，这些东西确实能让写代码这件事更靠谱。即使 AI 会让编程的范式逐渐改变，但是版本控制这样的思维永远是一个开发者必备的能力。后面这门课，我会认真听下去，多实践，实践结合理论才是这门课的精髓。
