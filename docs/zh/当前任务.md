## 第一部分：**必须做的**

| 项目                            | 为什么必须做                       | 你现在最小做法                                           | 备注                                           |
| ----------------------------- | ---------------------------- | ------------------------------------------------- | -------------------------------------------- |
| **测“当前系统可用上限”**               | 这是比赛真实价值，不先测清，PID/前馈/规划都会飘   | 不上来搞最大电流模式，只加一个**测试命令源**，能发固定/斜坡 `vx vy wz` 就够    | 先测“可用上限”，不是“绝对物理极限”                          |
| **给底盘加一个测试模式**                | 你现在头疼的根因不是控制本身，而是**没稳定测试入口** | 新增 `manual / auto / test` 三类命令源，测试模式只负责固定目标或斜坡目标  | 只改命令层，不先动控制核心                                |
| **把 `chassis_sub` 明确成统一命令对象** | 你已经走到这一步了，不命名清楚后面会反复混乱       | 可以先不改大结构，只把语义上认定为“底盘当前命令/参考”                      | 你现有 `speed_x speed_y speed_yaw halt` 已经具备雏形。 |
| **把参数分层**                     | 你现在“参数太多难管”，核心是**类型没分层**     | 只分 4 类：物理常量 / 硬件边界 / 控制参数 / 操作手感参数                | 你当前代码已经部分这么做了，可顺着整理。                         |
| **给 PID 研究设边界**               | 不然会被积分、微分、前馈、LADRC 一起拖走      | 近期只主攻：积分分离、抗饱和、D 项慎用/滤波、目标斜坡                      | 先把“比赛可用”做强                                   |
| **保留“全链路”思路，但缩成最小闭环**         | 你不是方向错，是一次想做太全               | 当前“全链路”只保留：命令源 → 规划 → 执行 → 电机控制 → 日志              | 不要同时做完整调参平台                                  |
| **建立最小日志集**                   | 没日志，调参与测试全靠感觉                | 只记录：目标速度、实际速度、控制输出、是否饱和、halt 状态                   | 不求全，只求能对比                                    |
| **建立 Git 轻流程**                | 不是没 Git，而是现在使用成本太高           | 只保留 3 条线：`race-stable` / `test-mode` / `ctrl-exp` | 先服务实验，不追求完美工作流                               |
| **和 ROS2 车约定“命令接口”，不要追求同实现**  | 两车方案不同，统一实现不现实，统一接口才现实       | 统一约定一套底盘命令语义：`vx vy wz halt frame source`         | 这样你纯 MCU 也不会被边缘化                             |
| **优先完成比赛主线，不跟风补 ROS2**        | 你现在时间不够，强补 ROS2 会直接稀释战力      | 把 ROS2 相关工作限定为“接口兼容意识”，不是立刻上栈                     | 这点非常重要                                       |

## 第二部分：**可暂缓**

| 项目                          | 为什么先缓一缓                  | 以后什么时候再做             |
| --------------------------- | ------------------------ | -------------------- |
| **完整 ROS2 学习与上车**           | 现在你最紧缺的是比赛底盘能力，不是中间件栈    | 底盘上限、测试入口、控制可复现都稳定后  |
| **micro-ROS + ROS2 双端联调体系** | 这是系统工程活，不适合在你当前控制迭代最忙时展开 | 等底盘控制链稳定、接口明确后       |
| **完整“数据调参平台”**              | 平台建设很吃时间，容易挤占控制本体进度      | 先靠最小日志 + 手动比对，后面再平台化 |
| **前馈完整建模补偿体系**              | 你自己已经看出来了，这不是加个项那么简单     | 等你先测清上限、跟踪误差来源后      |
| **LADRC 作为近期主力方案**          | 现在它更像研究题，不是比赛近期最稳妥主线     | 等 PID 体系做扎实后再对比      |
| **Ruckig 式完整速度规划实现**        | 现在更缺的是底层边界和测试通路，不是高级规划器  | 以后当你确实被轨迹平滑限制住时      |
| **大规模代码重构**                 | 现在最怕“为了整洁而失去可迭代性”        | 只做轻重构，别大拆            |
| **所有参数都运行时热调**              | 这会把测试系统复杂度拉爆             | 先挑少数关键参数可调，其余编译期固定   |
| **和队友完全统一技术方案**             | 两车定位不同，本来就不该完全一样         | 统一接口与数据语义就够了         |
