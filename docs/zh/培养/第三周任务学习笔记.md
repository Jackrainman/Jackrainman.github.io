# 第三周任务学习笔记

这份笔记对应 [第三周任务](第三周任务.md) 的前两项，目标不是把所有知识点堆满，而是把“检查时要能解释清楚”的主线整理出来。

## 1. C 语言从源文件到二进制文件的过程

### 1.1 总流程

一个 C 工程从源文件到最终可烧录/可执行文件，通常经历下面几步：

1. 预处理（Preprocess）
2. 编译（Compile）
3. 汇编（Assemble）
4. 链接（Link）
5. 格式转换（可选，嵌入式常见）

可以简单记成：

```text
.c/.h -> .i -> .s -> .o -> .elf -> .bin/.hex
```

### 1.2 每一步到底做了什么

#### 1.2.1 预处理

输入是 `.c` 和 `.h`，输出通常是 `.i`。

这一阶段主要做四件事：

1. 展开 `#include`
2. 替换 `#define` 宏
3. 处理条件编译，如 `#if`、`#ifdef`
4. 删除注释

例如：

```c
#define LED_PIN 5
#include "main.h"
```

经过预处理后，宏会被直接替换，头文件内容会被“拷贝展开”到当前源文件里。

这一步的本质是：把程序先整理成一个“更完整、更统一”的纯 C 文本，再交给真正的编译器前端处理。

#### 1.2.2 编译

输入是 `.i`，输出通常是 `.s` 汇编文件。

这一阶段编译器会完成：

1. 词法分析：把代码拆成关键字、标识符、运算符等 token
2. 语法分析：检查语法结构是否合法
3. 语义分析：检查类型、作用域、函数声明是否匹配
4. 优化：常量折叠、死代码删除、内联等
5. 生成目标平台对应的汇编代码

例如：

```c
sum = a + b;
```

编译后会变成 ARM、x86 或别的架构对应的汇编指令，不同 CPU 平台生成的汇编完全不同。

#### 1.2.3 汇编

输入是 `.s`，输出通常是 `.o` 目标文件。

汇编器会把汇编指令翻译成机器码，同时生成符号表、重定位信息等内容。  
这时得到的 `.o` 已经是“机器能识别的二进制片段”了，但还不是最终完整程序。

原因是：

1. 很多函数和变量地址还没有最终确定
2. 多个源文件之间还没拼起来
3. 库函数也还没并入最终映像

#### 1.2.4 链接

输入是多个 `.o` 文件和库文件，输出通常是 `.elf` 或 `.axf`。

链接器主要做三件事：

1. 合并各个目标文件的代码段、数据段
2. 解析符号引用
3. 完成重定位，确定最终地址

典型例子：

```c
// main.c
extern void led_init(void);

int main(void) {
    led_init();
}
```

```c
// led.c
void led_init(void) {
}
```

`main.c` 在编译时只知道“有个 `led_init` 函数”，但不知道它的真实地址；链接器会在链接阶段找到 `led.c` 里的定义，并把调用地址修正好。

对嵌入式来说，链接阶段还多了一个非常关键的东西：**链接脚本**。  
链接脚本会规定：

1. `FLASH` 从哪里开始
2. `RAM` 有多大
3. `.text`、`.data`、`.bss` 放到哪里
4. 中断向量表放到哪里

所以嵌入式程序不是简单“拼起来”就结束，而是要按芯片内存布局摆放。

#### 1.2.5 格式转换

这一步在 PC 上不一定显式出现，但在嵌入式里非常常见。

例如链接生成 `firmware.elf` 后，还会再生成：

1. `firmware.bin`
2. `firmware.hex`

原因是烧录工具、量产工具、Bootloader 不一定直接吃 `ELF`，很多场景更常用 `BIN/HEX`。

### 1.3 一个最常见的命令链

以 GCC 为例，可以理解成下面这样：

```bash
gcc -E main.c -o main.i
gcc -S main.i -o main.s
gcc -c main.s -o main.o
gcc main.o -o app.elf
```

嵌入式交叉编译时，只是把 `gcc` 换成目标平台编译器，例如：

```bash
arm-none-eabi-gcc -c main.c -o main.o
arm-none-eabi-gcc main.o -T stm32.ld -o firmware.elf
arm-none-eabi-objcopy -O binary firmware.elf firmware.bin
```

这里多出来的重点是：

1. 目标不是当前电脑，而是 ARM MCU
2. 需要链接脚本
3. 最终常常还要转成 `bin`

### 1.4 常见 C 编译器种类

按你现在容易接触到的场景，先记这些就够用：

#### 1.4.1 PC 常见编译器

1. GCC  
   Linux 上最常见，也常用于 MinGW、WSL 等环境
2. Clang/LLVM  
   前端诊断友好，工具链生态强
3. MSVC  
   Windows + Visual Studio 生态常见

#### 1.4.2 嵌入式常见编译器

1. `arm-none-eabi-gcc`  
   ARM Cortex-M 最常见的开源交叉编译器
2. ARM Compiler 5 / ARM Compiler 6  
   Keil MDK 常见，AC6 底层基于 LLVM
3. IAR Embedded Workbench 编译器  
   工业项目也很常见，优化和调试集成度较高
4. Keil C51 / C251  
   8051 这类老架构仍会见到

### 1.5 PC 编译器和嵌入式编译器的区别

这是检查时很容易被追问的点。

#### 1.5.1 目标平台不同

PC 编译器通常面向 `x86/x64`；  
嵌入式编译器通常面向 `ARM Cortex-M`、`RISC-V`、`8051` 等 MCU 架构。

#### 1.5.2 主机和目标往往不是同一个平台

PC 开发里，经常是“在 Windows/Linux 上编译，生成给 Windows/Linux 自己运行的程序”。  
嵌入式开发里，经常是“在 PC 上编译，但目标程序运行在单片机上”，这就叫**交叉编译**。

#### 1.5.3 运行环境不同

PC 程序背后通常有操作系统托底，有文件系统、进程、虚拟内存。  
MCU 程序很多时候是裸机或 RTOS，没有完整操作系统环境，程序必须自己完成：

1. 启动初始化
2. 中断向量表布置
3. 时钟配置
4. 堆栈和内存布局安排

#### 1.5.4 链接关注点不同

PC 更关注可执行文件格式、动态库、系统 ABI。  
嵌入式更关注：

1. `FLASH` / `RAM` 是否放对
2. 启动文件是否正确
3. 中断向量表地址是否正确
4. 程序大小是否超出芯片资源

#### 1.5.5 标准库实现不同

PC 上通常是完整 libc。  
嵌入式里常见 `newlib-nano`、microlib 这类“瘦身版”库，不然体积会太大。

#### 1.5.6 优化目标不同

PC 常更关注运行速度和通用兼容性。  
嵌入式还会非常关注：

1. 代码体积
2. RAM 占用
3. 实时性
4. 可预测性

### 1.6 检查时建议直接这样回答

如果学长问“C 文件怎么变成二进制”，可以先用一句话概括：

> C 源文件先经过预处理、编译、汇编、链接，最后在嵌入式里通常还会把 ELF 转成 BIN/HEX 用于烧录。

然后补三点：

1. 编译阶段把 C 变成目标平台汇编
2. 汇编阶段把汇编变成目标文件 `.o`
3. 链接阶段把多个 `.o`、库文件和启动文件按链接脚本拼成最终映像

## 2. 串口中断、DMA、基本定时器、通用定时器

### 2.1 串口中断“三种中断”先说明一个假设

这里我按 STM32 入门阶段最常见的说法整理为：

1. `RXNE`：接收寄存器非空中断
2. `TXE`：发送数据寄存器空中断
3. `TC`：发送完成中断

这是一个**带假设**的整理。  
有些资料会把 `IDLE` 空闲中断也作为常见重点讲，这时就把 `TC` 和 `IDLE` 一起补充理解即可。

### 2.2 串口中断的核心逻辑

串口本质上是一个串行外设，它和 CPU 之间不是“每次都主动轮询”，而是：

1. 平时外设自己工作
2. 发生特定事件时置位标志
3. 如果使能了对应中断，就进入中断服务函数
4. 在中断里读写数据寄存器或清标志

所以串口中断的主线是：

```text
外设事件发生 -> 状态标志置位 -> NVIC 响应 -> 进入 USARTx_IRQHandler -> 读写寄存器/清标志
```

### 2.3 三种常见串口中断分别是什么

#### 2.3.1 RXNE：接收寄存器非空

意思是：收到一个新字节，数据已经进了接收数据寄存器，CPU 可以来读了。

典型用法：

1. 单字节接收
2. 命令字解析
3. 简单串口回显

处理中最关键的一步是：**要把数据读出来**。  
因为不读，标志通常不会按预期释放，后续接收也可能出问题。

典型流程：

1. 串口收到 1 字节
2. 硬件把字节放入数据寄存器
3. `RXNE = 1`
4. 进入中断
5. 读取 `DR` / `RDR`
6. 把数据存到软件缓冲区

#### 2.3.2 TXE：发送数据寄存器空

意思是：发送数据寄存器空了，可以继续塞下一个字节。

典型用法：

1. 中断方式连续发送一串数据
2. 软件维护发送缓冲区，靠中断逐字节发送

注意：

`TXE` 表示“寄存器空了，可以继续装新数据”，**不等于一整帧已经从引脚发完**。

#### 2.3.3 TC：发送完成

意思是：不仅发送寄存器空了，而且最后一个字节也真正从移位寄存器发出去了。

典型用法：

1. RS485 半双工收发方向切换
2. 需要严格确认“最后一位也发完”时

这和 `TXE` 的区别非常重要：

1. `TXE` 看的是“还能不能继续装字节”
2. `TC` 看的是“线路上最后一位是不是也发完了”

### 2.4 如果资料里讲到 IDLE，中断要怎么理解

`IDLE` 是空闲中断。  
当串口接收完一段数据后，如果总线保持一个字符时间以上的空闲状态，就会触发。

它特别适合：

1. 不定长数据包接收
2. `DMA + 串口接收`
3. 一帧结束判定

所以在工程里，很多高效写法是：

```text
DMA 负责搬运数据
IDLE 中断负责告诉 CPU “这一帧差不多收完了”
```

### 2.5 DMA 是什么，为什么它比单纯中断更高效

DMA = Direct Memory Access，直接存储器访问。

它的本质是：**让硬件帮你搬数据，而不是让 CPU 一次一次搬**。

例如串口接收时：

1. 不用每来 1 个字节就进一次 `RXNE` 中断
2. DMA 可以直接把串口数据搬到内存数组里
3. CPU 只在“半满、满、空闲中断、传输完成”这类关键节点介入

所以 DMA 的优势是：

1. 降低中断频率
2. 减少 CPU 占用
3. 更适合高速、连续数据流

### 2.6 串口接收的三种典型方案

#### 2.6.1 轮询

CPU 一直检查标志位。

特点：

1. 最简单
2. 最占 CPU
3. 不适合高实时并发任务

#### 2.6.2 中断接收

每来一个字节触发一次中断。

特点：

1. 比轮询高效
2. 适合低到中等数据量
3. 数据量一大，中断会很多

#### 2.6.3 DMA 接收

DMA 自动搬运，CPU 只在关键节点处理。

特点：

1. 效率最高
2. 最适合连续数据流
3. 配置比普通中断更复杂

### 2.7 例程实现过程应该怎么讲

检查任务时，往往不是问你“能不能背定义”，而是让你说清楚例程从初始化到运行的过程。

#### 2.7.1 串口接收中断例程

一套最基础的实现流程通常是：

1. 使能 GPIO 和 USART 时钟
2. 配置 TX、RX 引脚复用功能
3. 设置波特率、字长、停止位、校验位
4. 使能 `RXNE` 中断
5. 在 NVIC 中打开 `USARTx_IRQn`
6. 在 `USARTx_IRQHandler` 中读取接收字节
7. 把字节写入数组/环形缓冲区
8. 主循环读取缓冲区并做业务处理

如果让你解释“为什么要在中断里读数据”，核心回答是：

> 因为接收事件已经发生，数据进了接收寄存器；中断函数里要及时把字节取走，避免丢数据，并把硬件事件转换成软件可处理的数据流。

#### 2.7.2 串口 DMA 接收例程

常见流程是：

1. 配置 USART
2. 配置 DMA 通道，方向设为“外设到内存”
3. 设置目标内存地址和搬运长度
4. 将 USART 接收和 DMA 链接
5. 启动 DMA 接收
6. 用 DMA 半传输/传输完成中断，或者 `IDLE` 中断来判断该处理哪段数据
7. 应用层从缓冲区读取有效数据

如果问“DMA 为什么快”，不要只说“因为硬件快”，更准确的回答是：

> 快的根本原因不是单次搬运速度神奇提高，而是减少了 CPU 频繁响应中断和逐字节搬运的参与度。

### 2.8 基本定时器和通用定时器的区别

以 STM32 常见系列为例，可以这样理解：

#### 2.8.1 基本定时器

典型代表：`TIM6`、`TIM7`

主要用途：

1. 纯定时
2. 周期中断
3. 作为 DAC 触发源

特点：

1. 结构简单
2. 没有输入捕获/输出比较通道
3. 通常不能直接输出 PWM

所以基本定时器最像“一个定时打铃器”。

#### 2.8.2 通用定时器

典型代表：`TIM2`、`TIM3`、`TIM4`、`TIM5`

主要用途：

1. 定时中断
2. 输入捕获
3. 输出比较
4. PWM 输出
5. 编码器接口（部分型号支持）

特点：

1. 功能比基本定时器丰富很多
2. 既能定时，也能和引脚打配合
3. 常用于舵机 PWM、电机 PWM、测频测脉宽

所以通用定时器不是“只能拿来计时”，而是一个更完整的时序外设。

### 2.9 定时器时间怎么计算

这是高频必问点。

定时器更新频率常用公式：

```text
f_update = 定时器时钟 / (PSC + 1) / (ARR + 1)
```

其中：

1. `PSC` 是预分频器
2. `ARR` 是自动重装值

例如：

1. 定时器时钟 = 72 MHz
2. `PSC = 7199`
3. `ARR = 9999`

则：

```text
f_update = 72,000,000 / 7200 / 10000 = 1 Hz
```

也就是每 1 秒更新一次。

### 2.10 基本定时器例程应该怎么讲

如果是“定时 500 ms 翻转一次 LED”这类例程，讲解顺序建议固定：

1. 先使能 TIM 时钟
2. 配置 `PSC` 和 `ARR`
3. 使能更新中断
4. 在 NVIC 中打开定时器中断
5. 启动定时器
6. 在中断服务函数里清更新标志并翻转 LED

一句话总结就是：

> 基本定时器的核心是先把计数周期配出来，再利用更新事件周期性进入中断函数执行任务。

### 2.11 通用定时器 PWM 例程应该怎么讲

比如“输出 PWM 控舵机”这类例程，主线通常是：

1. 使能 TIM 和 GPIO 时钟
2. 把对应引脚配置成复用输出
3. 设置 `PSC` 和 `ARR`，先把 PWM 周期定出来
4. 设置 `CCR`，决定占空比
5. 配置 PWM 模式
6. 使能通道输出
7. 启动定时器

如果问“为什么改 `CCR` 就能改占空比”，可以这样回答：

> 因为 `ARR` 决定一个 PWM 周期有多长，`CCR` 决定在这个周期里高电平持续多久，所以 `CCR / (ARR + 1)` 就对应占空比。

### 2.12 串口中断和外部中断的区别

这也是第三周任务原文里明确要求搞清楚的点。

#### 2.12.1 触发源不同

1. 串口中断来自 USART 外设内部事件  
   例如 `RXNE`、`TXE`、`TC`、`IDLE`
2. 外部中断来自 GPIO 引脚电平变化  
   例如上升沿、下降沿、双边沿

#### 2.12.2 关注对象不同

1. 串口中断处理的是“通信数据到了没有、发完没有”
2. 外部中断处理的是“引脚状态变了没有”

#### 2.12.3 中断处理内容不同

1. 串口中断里通常要读写数据寄存器
2. 外部中断里通常要判断哪条线触发、清挂起位、执行对应动作

#### 2.12.4 典型应用不同

1. 串口中断：串口接收命令、上位机通信、模块数据接收
2. 外部中断：按键触发、传感器边沿检测、霍尔/光电开关事件响应

#### 2.12.5 一个最简单的理解方式

可以直接记：

> 串口中断是“某个通信外设内部发生事件后通知 CPU”；外部中断是“某个引脚电平变化后通知 CPU”。

## 3. 检查时的高频追问与简答

### 3.1 为什么 `.h` 文件本身一般不单独编译

因为 `.h` 更多是声明集合，通常通过 `#include` 被并入 `.c`，最终参与编译的是展开后的源文件。

### 3.2 为什么一个工程会有很多 `.o`

因为每个 `.c` 文件通常会先独立编译成一个 `.o`，最后再统一链接。

### 3.3 为什么嵌入式经常需要链接脚本

因为必须明确代码段、数据段、中断向量表到底放到哪一段 `FLASH/RAM` 中。

### 3.4 为什么 DMA 适合串口接收大数据

因为它把“逐字节搬运”从 CPU 转移给硬件，减少中断次数和 CPU 占用。

### 3.5 为什么基本定时器不能像通用定时器那样方便输出 PWM

因为基本定时器本身主要面向“时间基准”和“更新事件”，没有通用定时器那套更完整的输出比较/PWM 通道能力。

### 3.6 为什么串口中断和外部中断不能混为一谈

因为两者的触发源、寄存器路径、应用目的都不同；一个偏通信外设事件，一个偏 GPIO 边沿事件。
