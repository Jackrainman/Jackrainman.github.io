# 第二阶段任务（系统整合阶段）

## 1. 阶段定位

本阶段目标是从“会做单点功能”升级为“会组织系统工程并完成模块整合”。

通过后至少应具备：

- 能使用 VSCode 和统一模板工程开展开发
- 能完成模块化分层、状态机设计和接口划分
- 能打通 I2C / SPI / OLED / Flash 等综合外设链路
- 能完成一次综合整合任务，并解释数据流和状态流

推荐节奏：3 到 5 周。任务仍可按周发布，但阶段通关优先于自然周数。

## 2. 阶段统一规则

- 进入条件：第一阶段通过
- 验收方式：面对面验收，按总则评分
- 本阶段重点在“整合、排错、结构”，不再只看单点现象
- FreeRTOS 可作为提升线或预研线，不强绑为全员同速硬门槛

---

## 节点 1：模板工程与迁移

### 本节点目标

1. 建立统一开发环境与模板使用流程
2. 理解模板目录、构建链和下载调试链路
3. 完成第一阶段小系统向模板工程的迁移

### 通关线

1. 按模板创建新工程并成功编译、下载
2. 将第一阶段小系统迁移到模板工程
3. 输出工程目录说明（模块职责 + 依赖关系）

### 提升线

- 建立统一脚本（编译 / 烧录 / 日志导出）
- 对工程目录和模块边界进行一次整理

### 预研线

- 一键构建或日志抓取脚本
- Debug / Release 双配置

### 验收问题（抽 2 到 3 个）

- 模板工程相比裸工程的核心收益是什么？
- 迁移时最容易出错的两个点是什么？
- 你如何保证工程可复现？

---

## 节点 2：模块化、状态机与接口设计

### 本节点目标

1. 完成输入层、业务层、输出层分离
2. 构建可扩展状态机
3. 控制模块耦合度，减少跨层乱调

### 通关线

1. 设计并实现状态机（至少 4 个状态）
2. 抽象命令接口（串口 / 按键统一到同一事件入口）
3. 输出模块接口文档（输入、输出、时序约束）

### 提升线

- 参数配置表或命令表驱动
- 关键状态切换增加保护条件

### 预研线

- 引入事件队列替代直接调用链
- 预研 FreeRTOS 的任务拆分方式

### 验收问题（抽 2 到 3 个）

- 如何避免主循环变成超级 `if-else`？
- 新增状态需要改动哪些最小位置？
- 模块边界如何定义“应该暴露什么接口”？

---

## 节点 3：综合外设链路

### 本节点目标

1. 打通 I2C / SPI / OLED / Flash 的基本链路
2. 建立状态、数据、显示之间的对应关系
3. 为综合大作业做模块级准备

### 通关线

1. I2C 或 OLED 最小跑通
2. SPI 或 Flash 最小跑通
3. 能说明数据从串口输入到存储、再到显示的基本路径

### 提升线

- 运行时间、消息数、剩余容量等状态显示
- 基本异常处理（非法输入、越界编号）

### 预研线

- 提前实现部分寒假综合任务
- 预研 FreeRTOS 多任务拆分

### 验收问题（抽 2 到 3 个）

- I2C 的起始、停止和 ACK 分别是什么？
- Flash 为什么要考虑擦除和地址管理？
- OLED 刷新为什么不能无脑高频刷？

---

## 节点 4：综合小系统收束

### 本节点目标

1. 完成一次真正的模块整合
2. 让新人从“会跟着例程做”走到“会合并、会排错、会解释”
3. 为第三阶段项目能力打地基

### 通关线

综合小系统至少包含：

- 串口输入
- 状态切换
- Flash 存储
- 列表查看
- 编号读取
- OLED 显示

### 提升线

- 完成参数持久化或日志分级
- 补充边界测试（超长输入、断电重启、越界索引）

### 预研线

- 用 FreeRTOS 拆分输入 / 控制 / 输出任务
- 为后续 CAN 接入预留接口层

### 验收问题（抽 2 到 3 个）

- 你是如何保证下一条数据不覆盖上一条的？
- 断电重启后，系统如何知道下一条该从哪里写？
- 当前系统里最容易出 bug 的环节是什么？

## 3. 阶段通关标准

满足以下全部条件：

1. 模板工程与开发流程可独立使用
2. 具备模块化拆分与状态机设计能力
3. 能打通 I2C / SPI / OLED / Flash 综合链路
4. 能完成综合整合任务并解释关键设计

## 4. 未过线补救路径

- 先补模板迁移和模块边界，再补综合外设
- 如果综合小系统未通过，先拆回模块级检查，不直接反复硬合并
- 下次验收优先看：数据流是否说得清、边界问题是否处理、临时小改是否能完成
