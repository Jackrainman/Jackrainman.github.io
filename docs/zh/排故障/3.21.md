每次进 HardFault，先把下面这些值抄下来或截图：

- `PC`
- `LR`
- `SP`
- `MSP`
- `PSP`
- `xPSR`
- `HFSR`
- `CFSR`
- `BFAR`
- `MMFAR`

这组值的意义我给你拆开讲。

**这些寄存器分别在告诉你什么**

- `PC`
    
    - 当前程序计数器
    - 你在 HardFault 里时，它大概率只会指向 `HardFault_Handler`
    - 所以它告诉你“现在人在哪”，但不一定告诉你“事故发生在哪”
- `LR`
    
    - 在 HardFault 里，这个值经常不是普通函数返回地址，而是一个特殊的 `EXC_RETURN`
    - 它最重要的作用是：告诉你异常发生前，应该看 `PSP` 还是 `MSP`
    - 比如这次 docs 里是 `0xFFFFFFFD`
    - 这说明 fault 发生在线程上下文，异常堆栈在 `PSP`
- `SP`
    
    - 当前栈指针
    - 但注意：你一旦进了异常处理函数，当前 `SP` 很可能已经不是“出事时线程正在用的栈”
    - 所以它要看，但不能盲信
- `MSP`
    
    - Main Stack Pointer
    - 中断/异常处理大多用它
    - 如果 fault 来自线程，通常它不是你最想看的那个栈
- `PSP`
    
    - Process Stack Pointer
    - FreeRTOS 线程/任务通常用它
    - 如果 `LR = 0xFFFFFFFD`，那你就该从 `PSP` 里读出“事故前快照”
- `xPSR`
    
    - 程序状态寄存器
    - 对新手来说不是第一优先级
    - 主要用于确认处理器状态、Thumb 状态、异常号等
    - 一般拿来辅助判断现场是否合理
- `HFSR`
    
    - HardFault Status Register
    - 它告诉你：这是不是“别的 fault 被升级成 HardFault”
    - 这次 docs 里 `HFSR = 0x40000000`
    - 说明这是 `FORCED`，也就是底层还有更具体的 fault 类型，不是纯黑盒
- `CFSR`
    
    - Configurable Fault Status Register
    - 这是最关键的 fault 分类寄存器之一
    - 它进一步告诉你：底层到底是 `MemManage`、`BusFault` 还是 `UsageFault`
    - 这次 `CFSR = 0x00008200`
    - 表示：
        - `PRECISERR`：精确总线错误，说明是哪次访问炸的，地址可信
        - `BFARVALID`：`BFAR` 里的地址有效，可以相信
- `BFAR`
    
    - BusFault Address Register
    - 如果是精确总线错误，它会告诉你“CPU 实际访问了哪个非法地址”
    - 这次是 `0x59531E45`
    - 这就是非常硬的证据
- `MMFAR`
    
    - MemManage Fault Address Register
    - 主要给内存管理 fault 用
    - 这次可以顺便记，但你们当前更关键的是 `BFAR`

一句话总结这组值的用途：

- `LR` 决定你去哪块栈里找现场
- `HFSR/CFSR` 决定你这次 fault 属于哪一类
- `BFAR/MMFAR` 决定“到底访问了哪个坏地址”

如果 fault 发生在线程态，而且 `LR = 0xFFFFFFFD`，那你就去看 `PSP` 指向的 8 个字：

```
stack[0] = r0;
stack[1] = r1;
stack[2] = r2;
stack[3] = r3;
stack[4] = r12;
stack[5] = stacked_lr;
stack[6] = stacked_pc;
stack[7] = stacked_xpsr;
```

这 8 个值里，最值得你重点盯的是下面几个。

**异常堆栈里这些值分别有什么意义**

- `r0`
    
    - 很多函数里，第一个参数或临时基址指针会放在这里
    - 这次它特别关键，因为 `node` 很可能就被编译器放进了 `r0`
    - 如果 `r0` 本身像垃圾地址，那就说明不是 `node->id` 逻辑错了，而是 `node` 早就坏了
- `r1`
    
    - 第二个参数或临时值
    - 这次 docs 里是 `0x204`
    - 它帮助你判断“当前正在处理哪个 ID 相关的逻辑”
- `r2 / r3`
    
    - 临时寄存器
    - 有时能看出数组基址、表地址、比较值
    - 不一定每次都关键，但有时能补齐现场
- `r12`
    
    - scratch register，临时寄存器
    - 一般不是第一优先级，但保留有价值
- `stacked_lr`
    
    - 这不是当前 HardFault 的 `LR`
    - 而是“出事那一刻，原代码本来打算返回到哪里”
    - 它有助于看调用链和前一层代码位置
- `stacked_pc`
    
    - 最关键
    - 这是真正发生 fault 的那条指令地址
    - 你要靠它把现场映射回源码
    - 这次已经映射到了 `New_G4/User/Bsp/can_list/can_list.c:413`
- `stacked_xpsr`
    
    - 出事瞬间的状态寄存器
    - 主要作为辅助校验

**第四步：结合你们这次的现场，怎么理解堆栈**

你们 docs 里已经有一组非常漂亮的证据：

```
r0         = 0x59531E41
r1         = 0x00000204
stacked_lr = 0x08007965
stacked_pc = 0x080079D2
BFAR       = 0x59531E45
```

再结合 `can_node_t` 的结构布局：

```
typedef struct can_node {
    void *can_data;          // +0
    uint32_t id;             // +4
    uint32_t id_mask;        // +8
    can_callback_t callback; // +12
    struct can_node *next;   // +16
} can_node_t;
```

你会发现：

- `BFAR = 0x59531E45`
- `r0 = 0x59531E41`
- 两者正好差 `4`

这意味着什么？

很像是：

- `r0` 里放着 `node`
- CPU 正在读 `node->id`
- 而 `id` 在结构体偏移 `+4`
- 所以访问地址正好是 `node + 4`

这就是为什么“看堆栈”这么重要：

- 如果你只看 `HardFault_Handler()`，你只知道“死了”
- 如果你看到了 `r0`、`BFAR`、`stacked_pc`
- 你就能推出“是在读 `node->id` 时炸的，而且 `node` 指针本身已经不对”

这比单纯说“can_list 有问题”高明很多。

**第五步：回到源码后，要看哪些局部变量**

现在既然已经落到了 `New_G4/User/Bsp/can_list/can_list.c:411` 到 `New_G4/User/Bsp/can_list/can_list.c:413` 这一段：

```
id = rx_header.Identifier;
node = table->table[id % table->len];

while ((node != NULL) && (node->id) != (id & node->id_mask)) {
    node = node->next;
}
```

你接下来就要看这一组局部变量。  
下面我不仅告诉你看什么，还告诉你为什么看。

- `recv_msg.hcan`
    
    - 作用：当前这帧是从哪一路 CAN/FDCAN 进来的
    - 为什么看：先分清是 `CAN1` 还是 `CAN2`
    - 意义：如果你不先分清总线，后面全在错的表里找
- `recv_msg.rx_fifo`
    
    - 作用：当前消息来自哪个 FIFO
    - 为什么看：帮助确认接收路径是不是你预期的那条 ISR/队列链
    - 意义：有时同一 CAN 不同 FIFO 注册不同消息
- `rx_header.Identifier`
    
    - 作用：当前收到的 CAN ID
    - 为什么看：这就是这次消息的“目标号”
    - 意义：后面 `id % table->len` 就靠它选桶
- `rx_header.IdType`
    
    - 作用：标准帧还是扩展帧
    - 为什么看：它决定你走 `STD_ID_TABLE` 还是 `EXT_ID_TABLE`
    - 意义：如果表都选错了，后面的 `node` 当然可能异常
- `can_received`
    
    - 作用：内部枚举，表示当前被识别成 `can1_selected` / `can2_selected` / `can3_selected`
    - 为什么看：确认 `recv_msg.hcan` 被代码正确映射到了哪个 `can_table[]`
    - 意义：它是“总线对象 -> 软件表索引”的桥
- `table`
    
    - 作用：当前选中的那张 ID 哈希表
    - 为什么看：确认你现在到底在查标准表还是扩展表
    - 意义：如果 `table` 本身是坏的，那问题比 `node` 还早
- `table->len`
    
    - 作用：桶数量
    - 为什么看：`id % table->len` 要靠它；而且也能看出表是否初始化正常
    - 意义：如果 `len` 异常为 0 或乱值，那 fault 方向就不同了
- `table->table`
    
    - 作用：桶数组的基地址
    - 为什么看：确认桶数组本体是不是个合理的 SRAM 地址
    - 意义：如果连这个指针都坏了，就不是单个节点坏，而是表结构坏
- `table->table[id % table->len]`
    
    - 作用：这次 ID 会落到哪个桶
    - 为什么看：这是你本次真正要取出来的 `node`
    - 意义：能直接判断“出事的是哪个桶”
- `node`
    
    - 作用：当前链表节点指针
    - 为什么看：这次最像就是它坏了
    - 意义：如果 `node` 不是正常 `0x200xxxxx` 一类的 RAM 地址，那就已经非常可疑
- `node->id`
    
    - 作用：当前节点注册的 ID
    - 为什么看：代码正在比较它
    - 意义：但前提是 `node != NULL` 且地址合理，否则别乱展开
- `node->id_mask`
    
    - 作用：用于掩码匹配
    - 为什么看：如果不是精确 ID，而是掩码匹配，得确认比较逻辑没跑偏
    - 意义：辅助理解 while 条件
- `node->next`
    
    - 作用：链表后继节点
    - 为什么看：如果当前桶里不止一个节点，代码会继续往后走
    - 意义：能帮助判断是“首节点坏了”还是“链表中间坏了”

**这里有一个很重要的调试习惯**

- 如果 `node == 0x0`
- 不要展开 `node->id`、`node->next`

原因 docs 已经提到过：

- 调试器可能把 `0x00000000` 地址上的向量表误解释成 `can_node_t`
- 你会看到一些看起来很离谱但其实是“假结构体”的内容
- 这会把你带偏

所以你的规则应该是：

- `node == NULL`：只承认它是空，不展开
- `node != NULL` 且地址像正常 RAM：再展开字段

**第六步：为什么“局部变量 + 堆栈”要一起看**

因为这两者解决的问题不一样。

- 堆栈告诉你：CPU 当时在执行哪一条指令，参与运算的寄存器值是什么
- 局部变量告诉你：这条源码语句对应的业务上下文是什么

举个你们当前这个案子的理解方式：

- `stacked_pc` 告诉你 fault 落在 `can_list.c:413`
- `r0` 和 `BFAR` 告诉你炸在读 `node->id`
- `rx_header.Identifier` 和 `can_received` 告诉你这次到底是哪一路 CAN、哪个 ID 的消息触发了这次遍历
- `table->table[id % table->len]` 和 `node` 告诉你到底是哪一个桶里的节点坏了

这就从“程序炸了”变成了：

- 哪个函数炸
- 哪一行炸
- 哪个对象坏
- 处理的是哪一路总线
- 对应的是哪个消息 ID

这就已经非常接近 root cause 了。

**第七步：看完 HardFault 堆栈之后，下一步怎么继续**

你学长说“先看堆栈，再继续”，这个“继续”不是随便继续跑，而是进入第二阶段：

- 不再问“哪里炸了”
- 改问“谁更早把它写坏了”

因为按你们当前记录，`can_list.c:413` 很可能只是第一个碰到坏指针的地方，不一定是写坏它的地方。

所以后面最合理的动作是：

- 对 `can_table` 的有效桶下数据断点
- 先抓“谁改了桶指针”
- 如果桶指针没变，再抓“谁改了节点内容”

优先盯这些对象：

- `can_table[0]->id_table[0].table[1]`
- `can_table[0]->id_table[0].table[2]`
- `can_table[0]->id_table[0].table[3]`
- `can_table[1]->id_table[0].table[1]`
- `can_table[1]->id_table[0].table[2]`
- `can_table[1]->id_table[0].table[3]`

为什么盯它们？

- 因为你们当前是三轮配置
- 有效桶主要就落在这些位置
- fault 又明显和 `node` 指针有关
- 所以先盯“桶里装的节点地址有没有被改”

**你调试时可以按这个固定节奏来**

1. 程序进 HardFault
2. 先抄 `LR / PSP / HFSR / CFSR / BFAR`
3. 再读 `PSP[0..7]`，拿到 `stacked_pc`
4. 把 `stacked_pc` 映射回源码
5. 在那一行看 `id / can_received / table / node`
6. 判断是“这一行逻辑错”还是“这一行只是撞上了坏数据”
7. 如果是坏数据，再去下 watchpoint 抓写坏者

**你现在最该建立的理解框架**

以后看到 HardFault，不要先想“我要改哪里”，而要按这三个问题来问：

- fault 是哪一类？  
    看 `HFSR/CFSR`
    
- CPU 实际访问了哪个坏地址？  
    看 `BFAR/MMFAR`
    
- 出事前最后一条指令是什么？  
    看 `stacked_pc`
    

如果再往业务层走一步，就是：

- 这条指令当时在处理哪个对象、哪一路消息、哪个 ID？  
    看 `r0~r3` + 源码局部变量

这样你是在“读证据”，不是在“猜答案”。

按目前 `Docs/HARDFAULT_DEBUG_RECORD.md` 里的证据，这条链已经很清楚了：

- 底层实质是线程态 `BusFault`

- 真正落点在 `New_G4/User/Bsp/can_list/can_list.c:413`
